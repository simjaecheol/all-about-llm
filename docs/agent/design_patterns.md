# 에이전트 설계 패턴 (Agentic Design Patterns)

단순한 일방향성 파이프라인보다 강력한 성능을 내기 위해 에이전트 시스템 설계 시 자주 사용되는 패턴들입니다.

## 1. Reflection (반성/검토) 패턴
모델이 생성한 결과물을 스스로 혹은 다른 모델이 검토하여 품질을 높이는 패턴입니다.
- **Generator**: 답변 초안을 생성합니다.
- **Reflector/Critic**: 초안의 오류나 개선점을 찾아 피드백을 줍니다.
- **Refinement**: 피드백을 반영하여 최종 답변을 완성합니다.

## 2. Tool Use (도구 활용) 패턴
에이전트가 자신이 알지 못하는 정보나 수행할 수 없는 작업을 위해 외부 도구를 활용하는 패턴입니다.
- **Intent Detection**: 사용자의 질문에서 도구 사용 의도를 파악합니다.
- **Argument Extraction**: 도구 실행에 필요한 파라미터를 추출합니다.
- **Observation Integration**: 도구 실행 결과를 다시 문맥에 통합합니다.

## 3. Planning & Execution (계획 및 실행) 패턴
복잡한 문제를 한 번에 해결하지 않고, 먼저 전체 계획을 세운 뒤 하나씩 실행하는 구조입니다.
- **Planner**: 전체 미션을 하위 단계들로 쪼개어 리스트업합니다.
- **Executor**: 계획된 순서대로 작업을 수행합니다.
- **Re-planner**: 실행 결과가 예상과 다를 경우 중간에 계획을 수정합니다.

## 4. Self-Correction (자기 수정)
코드 실행 에러나 논리적 모순이 발견되었을 때, 에이전트가 오류 메시지를 보고 스스로 수정안을 제시하는 패턴입니다. 특히 **Code Interpreter**를 사용하는 에이전트에서 필수적인 패턴입니다.

## 5. Routing (라우팅)
입력된 질문의 특성에 따라 서로 다른 전문 에이전트나 워크플로우로 작업을 전달하는 패턴입니다. 비용과 성능 사이의 균형을 맞추는 데 효과적입니다.
- **Simple Routing**: 특정 키워드나 분류기를 통해 전달.
- **Dynamic Routing**: LLM이 질문을 분석하여 가장 적합한 전문가 에이전트를 선택.

---

### 패턴 적용 시 주의사항
- **Infinity Loop**: 에이전트가 서로를 계속 비판하거나 수정을 끝내지 못하는 상황을 방지하기 위해 최대 반복 횟수를 설정해야 합니다.
- **Latent Error Accumulation**: 중간 단계의 작은 오류가 최종 결과에서 크게 번질 수 있으므로 각 단계의 검증이 필요합니다.
